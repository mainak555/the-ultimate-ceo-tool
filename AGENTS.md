## Key Rules

1. **Business logic lives in `server/services.py`** — views are thin controllers
2. **Validation lives in `server/schemas.py`** — returns clean dicts or raises `ValueError`
3. **MongoDB access lives in `server/db.py`** — singleton connection, no ORM
4. **Model catalog lives in `agent_models.json`** — model names are the UI key and are always displayed ascending
5. **AutoGen runtime code belongs in root `agents/`** — keep provider/client logic out of `server/`
6. **No Django ORM** — `DATABASES = {}`, sessions use signed cookies
7. **`APP_SECRET_KEY`** gates write access — HTMX requests send it as `X-App-Secret-Key`
8. **All env vars** read via `os.getenv()` with sensible defaults
9. **Templates** use HTMX partials pattern: full page loads `config.html`, subsequent interactions swap partials into `#main-content` or `#sidebar-list`
10. **SCSS** compiled by django-compressor + django-libsass
11. **No test suite yet** — planned for a future phase
12. **Project deletion safety**: never cascade delete chats when deleting a project. If any chat sessions exist for a project, deletion must be blocked with a clear error message.
13. **Common layer remains common**: global/shared modules (for example `server/static/server/js/app.js`) may contain only cross-feature utilities and hooks.
14. **Feature ownership is mandatory**: Home, Project Config, Trello, and Jira implementations must stay separated in HTMX templates, JS modules, views, and services. Avoid adding feature-specific logic to shared files.
15. **Provider registry is required for exports**: shared modules must use `server/static/server/js/provider_registry.js` (`window.ProviderRegistry`) instead of hardcoding provider names or provider-specific window globals.
16. **Reusable export modal pattern is mandatory**: all export providers (Trello, Jira, PDF, n8n, future) must use `window.ExportModalBase` (`server/static/server/js/export_modal_base.js`) as the shared modal shell. The base owns the overlay, header ("Export to {label}"), 70/30 split layout, right-pane reference loading, and footer buttons. Providers implement an adapter object; never build a modal overlay DOM structure inside a provider file.
17. **Visual consistency is mandatory across pages**: destructive controls (delete buttons/icons), color-token usage, spacing rhythm, and modal typography must match shared patterns defined in SCSS and docs; provider-specific theming is additive, not divergent.
18. **Extension skills are required for new providers**: follow `.agents/skills/export_popup_base/SKILL.md`, `.agents/skills/export_provider_adapter/SKILL.md`, `.agents/skills/ui_consistency_guardrails/SKILL.md`, `.agents/skills/scss_style_consistency/SKILL.md`, `.agents/skills/markdown_viewer_reuse/SKILL.md`, `.agents/skills/hierarchical_export_items/SKILL.md` (when items can nest), `.agents/skills/observability_logging/SKILL.md`, and `.agents/skills/export_agents_sync/SKILL.md` before implementing a new export provider.
19. **SCSS consistency is mandatory**: all styling changes must follow `docs/scss_style_guide.md` and must not introduce hardcoded color values when shared tokens exist.
20. **Markdown rendering must be reusable**: shared markdown rendering belongs in `server/static/server/js/markdown_viewer.js`; Home, Trello popup, Jira popup, and future providers must consume this common module instead of duplicating parsers.
21. **Jira export_agents are per-type**: for Jira, `export_agents` is scoped to each project type (`integrations.jira.software.export_agents`, etc.). There is no global `integrations.jira.export_agents` field. Validation, normalization, and template rendering must all read from the per-type config.
22. **Textarea fields in config forms must include a `<small class="form-hint">` below them** describing the field's purpose in plain language. The hint must be specific to the field's integration and type (e.g. "Prompt used by the extraction agent to parse the discussion into Jira Software issues."). Never leave a textarea without a hint.
23. **Nested fieldset indentation must be uniform across all nesting levels**: both L1 (`.form-group--nested`) and L2 (`.form-group--nested-l2`) use `margin-left: $space-md` and `padding-right: $space-sm`. Do not use `$space-lg` or any larger value for L2, and do not add `margin-top` to nested fieldsets — vertical rhythm is provided by the preceding element's `margin-bottom`. Missing `padding-right` causes textarea scrollbars and inputs to clip at the section edge. See `docs/scss_style_guide.md` §"Section Fieldsets (Config Form)" rules 4–6.
24. **`export_modal_base.js` is the only modal shell**: all export providers call `window.ExportModalBase.open(ctx, adapter)` and implement the adapter interface defined in `.agents/skills/export_popup_base/SKILL.md`. No provider file may build an overlay DOM with `overlay.innerHTML = ...` for the modal wrapper.
25. **Export modal context must include `projectId`**: the context object passed to `ProviderRegistry.openExportModal()` must always carry `{provider, sessionId, discussionId, secretKey, csrfToken, projectId}`. A missing `projectId` is a defect that must be fixed in `home.js`.
26. **Jira backend separation is mandatory**: `server/jira_service.py` is the shared facade only. Type-owned logic must live in `server/jira_software_service.py`, `server/jira_service_desk_service.py`, and `server/jira_business_service.py`.
27. **Jira frontend adapter separation is mandatory**: shared Jira export modal lifecycle logic belongs in `server/static/server/js/jira_adapter_factory.js`; type-owned provider registration belongs only in `jira_software.js`, `jira_service_desk.js`, and `jira_business.js`.
28. **Jira layering skill is required for Jira refactors**: follow `.agents/skills/jira_layer_separation/SKILL.md` before changing Jira services or Jira export adapters.
29. **Jira Software export modal metadata flow is mandatory**: project-specific dropdown options (issue type, priority, sprint, existing issue catalog) must come from `GET /jira/<session_id>/metadata/software/?project_key=<key>` and must be fetched via `jira_adapter_factory.js` after destination project selection.
30. **Jira Software global Sprint cascade is mandatory**: destination-level Sprint selector defines default values for all issue rows and must overwrite all rows whenever the global selector changes; row-level selectors remain editable for per-issue overrides after propagation.
31. **Jira Software fallback behavior is mandatory**: if project metadata cannot be loaded, the modal must remain usable by falling back to default issue type/priority options and minimal Sprint option (`Backlog`), with a non-blocking status message.
32. **Jira export label contract is mandatory**: `export_modal_base.js` header title uses adapter `label`, while export button text may use adapter `pushLabel` override. Jira Software currently uses header label `Jira` and push label `Jira Software`.
33. **Export popup add-action button style is mandatory**: left-pane create/add actions for cards/issues/items must use shared class `export-modal__context-add-btn` so all providers keep the same contextual add-button color treatment.
34. **Export popup card background contract is mandatory**: editable item cards across providers must use token-derived light panel backgrounds (for example `lighten($color-bg, 1.5%)`) with shared border/radius rhythm; avoid provider-specific hardcoded card background colors.
35. **Export popup item heading count badge is mandatory**: editable item section headings must use concise labels with a shared count badge (`export-modal__count-badge`), e.g. `Cards <count>`, `Issues <count>`, so counts are shown consistently across providers.
36. **Jira connection status message contract is mandatory**: the Jira export modal connection row must display `{Jira type label} Connected` on success (for example `Jira Software Connected`) and a clear type-scoped error message on failure.
37. **Jira metadata option deduplication is mandatory**: issue-type/priority dropdown options must be deduplicated by display label before rendering so duplicate labels (for example `Epic`) are never shown twice in issue cards.
38. **Project Config readonly markdown contract is mandatory**: readonly Objective, assistant system prompts, team selector system prompt, and integration extraction prompts (Trello and Jira types) must render via `window.MarkdownViewer.render()` using markdown target containers, not plain `<p>`/`<pre>` text blocks.
39. **Project Config readonly integrations parity is mandatory**: when integrations are enabled, readonly view must render Trello plus each enabled Jira type section (`software`, `service_desk`, `business`) with type-scoped readonly fields and extraction prompt visibility.
40. **Hierarchical export items use temp_id + parent_temp_id only**: any export provider whose items can nest (currently Jira Software, plus future providers) must persist parent linkage via `temp_id` + `parent_temp_id` and must NOT store `depth_level`. Depth is derived at render and push time. See `.agents/skills/hierarchical_export_items/SKILL.md` for the full data, render, and push contracts.
41. **Jira Software left-pane cascade is Project + Sprint only**: the Epic dropdown has been removed from the export modal. Parent linkage is expressed via the issue tree (`parent_temp_id`), not via a global Epic selector. The Sprint selector must always include a `Backlog` option whose value is the empty string.
42. **Jira Software push must be BFS with `temp_to_key` mapping**: `push_issues_software` walks roots first then breadth-first, populates `temp_to_key` as each issue is created, and resolves child `parent_temp_id` to a real Jira key before sending `fields.parent`. If an item carries `existing_issue_key`, push must update that existing Jira issue using the edited card fields, map `temp_to_key[temp_id] = existing_issue_key`, and continue BFS so descendants link correctly. A child whose parent failed to create must record a warning and be created as a root, never abort the batch. Result entries must echo `temp_id`.
43. **Jira Software sprint assignment via Agile API**: a non-empty `sprint` value triggers `POST /rest/agile/1.0/sprint/{id}/issue` AFTER the issue is created. An empty `sprint` value means Backlog and the Agile API call must be skipped entirely (the issue lands in backlog by default). Sprint failures are per-issue warnings, never batch failures. Issue types `Epic` and `Sub-task` are non-sprintable and must skip the Agile call with a warning.
44. **Observability contracts are mandatory for all I/O code**: every Python module that performs HTTP, MongoDB, file, or LLM I/O must declare `logger = logging.getLogger(__name__)`, never use `print()`, never log secrets (API keys, OAuth tokens, `Authorization`/Basic-auth, passwords, `X-App-Secret-Key`, Trello `key=`/`token=` query params), and propagate `X-Request-ID` via `RequestIdMiddleware`. Event names use dotted snake_case scoped by layer (e.g. `project.created`, `tracing.enabled`). Before adding a new HTTP client, service module, MongoDB path, agent-runtime entry, or model client wrapper, read [`.agents/skills/observability_logging/SKILL.md`](.agents/skills/observability_logging/SKILL.md) for the full logger, event-name, redaction, span-payload, and tracing-decorator contracts. Full reference: [docs/observability.md](docs/observability.md).
45. **OpenTelemetry is the project-wide tracing standard**: all I/O layers emit spans into a single trace per request via `@traced_function` / `traced_block` from `core/tracing.py`, plus three category-gated auto-instrumentation toggles — `OTEL_INSTRUMENT_HTTP` (Django + `requests`, default **on**), `OTEL_INSTRUMENT_MONGO` (pymongo, default **off** — high-volume DB spans), `OTEL_INSTRUMENT_AGENTS` (AutoGen event-log → span bridge, default **on**). The OTLP exporter is **pluggable** — Langfuse is wired today via `_build_langfuse_exporter()`; swapping backends is a one-function change in `_build_exporter()`. Span payloads are always set through `set_payload_attribute()` so redaction (`redact_payload()`) and truncation (`OTEL_MAX_PAYLOAD_BYTES`, default 32 KB) run uniformly. Console span output is opt-in via `OTEL_CONSOLE_EXPORTER` (`off` / `error` / `all`); `LOG_LEVEL=DEBUG` upgrades the default to `all`. Treat the OTLP backend as PII-bearing (Trello card descriptions, Jira issue summaries are forwarded as payloads).
46. **Model endpoint fallback contract is mandatory**: in `agents/factory.py`, endpoint resolution order is JSON `endpoint` first, then `{PROVIDER_UPPER}_API_URL` env fallback. API key resolution remains `{PROVIDER_UPPER}_API_KEY`. Azure providers (`azure_openai`, `azure_anthropic`) must still fail fast if endpoint is missing after fallback.
47. **Export schema documentation contract is mandatory**: every export popup provider must document its required extracted JSON schema in `README.md` (admin-facing) and in a provider-specific integration doc under `docs/` (developer-facing). Any schema or push-payload behavior change must update both docs in the same PR. Follow [docs/export_schema_contracts.md](docs/export_schema_contracts.md), [`.agents/skills/export_popup_base/SKILL.md`](.agents/skills/export_popup_base/SKILL.md), and [`.agents/skills/export_provider_adapter/SKILL.md`](.agents/skills/export_provider_adapter/SKILL.md).
48. **Shared HTTP span helper is mandatory for integrations**: outbound provider clients (Trello, Jira, and future integrations) must use `core/http_tracing.py` (`instrument_http_response`) for span enrichment and error status/details. Name response handlers meaningfully (for example `_handle_api_response`), never generic `_check`. Do not implement provider-local span plumbing helpers (`_current_span`, `_enrich_span`, `_mark_span_error`). `OTEL_HTTP_LOG_BODY` (default on) controls whether 2xx outbound calls capture `input.value`/`output.value`; non-2xx branches, including graceful fallback paths that continue execution, must still call `instrument_http_response(..., detail=...)` and always capture payloads/error details.
49. **MCP tool scope is per-agent + project-level shared**: each assistant agent stores `mcp_tools` ∈ {`none`, `shared`, `dedicated`}. `dedicated` requires a non-empty `mcp_configuration` (`{"mcpServers": {...}}`) on the same agent. `shared` requires a non-empty project-level `shared_mcp_tools` (`{"mcpServers": {...}}`). Validation must reject `shared` agents when `shared_mcp_tools` is empty. SSE transport is **not supported** (deprecated upstream); only stdio (`{command,args,env}`) and streamable HTTP (`{transport:"http",url,headers}`) are accepted. See [docs/mcp_integration.md](docs/mcp_integration.md) and [`.agents/skills/mcp_tool_integration/SKILL.md`](.agents/skills/mcp_tool_integration/SKILL.md).
50. **MCP runtime wiring lives in `agents/mcp_tools.py`**: `agents/team_builder.py` calls `resolve_mcp_servers_for_agent()` + `build_mcp_workbenches()` and passes the resulting workbenches to `AssistantAgent(workbench=..., reflect_on_tool_use=True)`. `reflect_on_tool_use=True` is mandatory for every agent that receives an MCP workbench — without it, AutoGen skips the second LLM call after tool execution and returns a raw `ToolCallSummaryMessage` that is neither rendered in the SSE chat nor captured as an `LLMCallEvent` span. Lifecycle is owned by `agents/runtime.py`: `evict_team()` must call `close_session_workbenches(session_id)`. Never construct `McpWorkbench` from `server/`. **Thread-pool safety**: `close_session_workbenches` uses `asyncio.get_running_loop()` (not the deprecated `get_event_loop()`) so it works correctly from Django thread-pool threads; never-started workbenches that raise `TypeError` on `stop()` are silently skipped.
51. **MCP secrets must never be logged or traced**: `args`, `env`, `headers`, and `url` query strings of MCP server entries are credentials by policy. Logs and OTel span attributes may include only `scope`, `server_count`, `server_names`, and a stable `fingerprint` hash. Use `set_payload_attribute()` + `redact_payload()` for any payload that must be carried.
52. **MCP deployment topology is governed by `deployments/`**: `standalone/` bundles Node in the app image (single-container hosts); `compose/` and `k8s/` use a Python-only app container plus a Node-based **mcp-gateway** sidecar exposing each MCP server over streamable HTTP. New deployment targets must follow the same split and document MCP transport reachability.
53. **MCP secrets-by-reference is mandatory for all credential material**: never embed raw API keys, tokens, or passwords directly in `shared_mcp_tools` or per-agent `mcp_configuration`. Store them under the project-level `mcp_secrets` dict (`{KEY: value}`) where `KEY` matches `^[A-Z][A-Z0-9_]*$` (UPPER_SNAKE), and reference them from any string value in `mcpServers` entries (including `args`, `env` values, `url`, `headers` values) using the `{KEY_NAME}` placeholder. `validate_project()` rejects any placeholder that has no matching `mcp_secrets` key. Substitution happens **only** at runtime in `agents/mcp_tools.py::_substitute_secrets()` immediately before `McpWorkbench` construction; `server/` code never sees substituted values.
54. **MCP secrets follow the SECRET_MASK contract**: `normalize_project()` returns `{KEY: SECRET_MASK}` for every stored secret so the form re-renders password inputs without leaking values; `_restore_masked_secrets()` swaps `SECRET_MASK` back to the DB value on save (keys absent from the submitted payload are treated as user deletions and dropped). Readonly view (`config_readonly.html`) MUST NOT render secret values — only an optional `🔒 N secrets configured` count badge listing key names.
55. **MCP secret fingerprint is computed over placeholders, not values**: `build_mcp_workbenches()` computes `_server_fingerprint()` over the placeholder `mcpServers` dict (pre-substitution), so the OTel `fingerprint` attribute stays stable across secret rotations and never carries credential material. Logs and span attributes for MCP must continue to expose only `scope`, `server_count`, `server_names`, `fingerprint`.
56. **Key/value form pattern is the only blessed UI for repeating KEY → value rows**: MCP Secrets, Trello Custom Fields, and any future provider list-of-pairs control (Jira custom fields, n8n inputs, header maps) must follow [`.agents/skills/key_value_form_pattern/SKILL.md`](.agents/skills/key_value_form_pattern/SKILL.md) — `1fr 2fr auto` grid, `+ Add` button in the subsection header, `chat-session-item__delete` × button per row, JS reindex on remove, and (for sensitive values) `<input type="password">` with the SECRET_MASK round-trip.
57. **Datetime storage standard is mandatory**: all datetime fields in MongoDB must be stored as BSON Date objects (`datetime.now(timezone.utc)`) — never as ISO strings or bare `HH:MM` strings. Use `_utc_now()` from `server/services.py` to produce values. Read paths must coerce BSON datetimes to ISO strings via `_coerce_dt_to_iso()`. This includes `project_settings.created_at`, `project_settings.updated_at`, `project_settings.integrations.trello.token_generated_at`, `chat_sessions.created_at`, `chat_sessions.discussions[].timestamp`, `chat_sessions.agent_state.saved_at`, `chat_sessions.discussions[].exports.*.updated_at`, `chat_sessions.discussions[].exports.*.last_push.pushed_at`, `chat_attachments.uploaded_at`, and `chat_attachments.bound_at`. Templates must wrap values in `<time class="local-time" data-utc="ISO">` elements; `window.renderLocalTimes()` in `app.js` converts to browser-local display. JS live injections use `new Date().toISOString()` + `window.renderLocalTimes()`. See [`.agents/skills/datetime_storage/SKILL.md`](.agents/skills/datetime_storage/SKILL.md) for the full contract and new-field checklist.
58. **Active run coordination is Redis-backed and fail-fast**: chat run ownership must be coordinated by `agents/session_coordination.py` using one lease per `session_id` plus heartbeat and cross-instance cancel signaling. `POST /chat/sessions/<id>/run/` must fail fast when Redis is unavailable and must not enter `running`. MongoDB remains the durable source for `chat_sessions.agent_state` and discussion history used for resume.
59. **Run lifecycle refactors require the active-session skill**: before changing chat run/start/stop/lease/cancel behavior, follow [`.agents/skills/active_session_coordination/SKILL.md`](.agents/skills/active_session_coordination/SKILL.md).
60. **Chat attachments are session-scoped by storage key**: all attachment blobs must use the flat key `sessions/{session_id}/attachments/{attachment_id}/{filename}` — no `message_id` segment, no project prefix. This key is assigned at upload time and is permanent (no rename at bind). All attachment lookup/bind/read operations must validate `session_id` equality before returning metadata or bytes.
61. **Session delete must clean attachment bytes, Redis cache, and metadata**: deleting a chat session must call `purge_session_attachment_cache(session_id)` first (purges Redis text-cache keys), then delete all blob objects under `sessions/{session_id}/`, then delete all `chat_attachments` metadata rows — in that order in the same operation.
62. **Attachment text extraction is lazy and Redis-backed**: `extracted_text` and `extraction_status` must never be stored in MongoDB. `server/attachment_service.py::build_attachment_context_block` checks Redis for cached text (`{REDIS_NAMESPACE}:attachment:{session_id}:{attachment_id}:text`) first; on miss it downloads from blob, extracts, writes to Redis with `REDIS_ATTACHMENT_TTL_SECONDS` TTL (default 24 h), and returns the full text. **Extraction failures (blob download error, pypdf error, etc.) are never written to Redis** — the next run retries extraction. Genuinely empty documents (e.g. scanned PDFs with no text layer) are cached normally to avoid repeated blob downloads. Supported extractable types: PDF (up to 50 pages), DOCX, PPTX (up to 50 slides), XLSX/XLS (all sheets, tab-separated rows), CSV (up to 200 rows), TXT, MD, JSON. Images are excluded from this path and loaded separately by `load_images_for_agents()`. Redis is provided by `agents.session_coordination.get_redis_client()` — `server/` code must never create a second Redis pool.
63. **Vision images are passed as `MultiModalMessage` to agents**: `server/views.py::_build_agent_task_for_run(task_text, session_id, attachment_ids)` downloads image bytes from blob via `load_images_for_agents()`, wraps each as `autogen_core.Image(PIL.Image.open(...))`, and returns `MultiModalMessage(content=[task_text, img1, img2, ...], source="user")` for `team.run_stream`. If PIL/autogen_core imports fail or all image downloads fail, it falls back to returning the plain task string — the agent run must never be blocked by image processing errors.
64. **Chat history attachment rendering contract is mandatory**: persisted chat messages must render attachment rows under message markdown content. Image attachments must render thumbnails; non-image attachments render filename links.
65. **HITL markdown + attachment parity is mandatory**: human gate notes must be markdown-rendered in live and persisted history, and the gate input surface must support the same attachment interactions as main chat input (attach button, drag/drop, paste).
66. **Chat attachment implementation skill is required**: before changing chat attachment upload/bind/render/delete behavior, follow [`.agents/skills/chat_attachment_workflow/SKILL.md`](.agents/skills/chat_attachment_workflow/SKILL.md).
67. **Azure attachment auth must use container SAS URL**: attachment storage auth must use `AZURE_STORAGE_CONTAINER_SAS_URL` (container-scoped SAS URL with query token), not account connection strings. SAS tokens should be least-privilege and short-lived where operationally possible.
68. **Single-assistant chat mode is mandatory**: when a project has exactly one assistant agent, the configuration must run in chat mode with Human Gate enabled and Team Setup hidden in the UI. Selector routing is invalid for one assistant. In this mode, runs pause after every assistant turn and continue until the human explicitly selects Stop (no max-iteration completion gate). Empty Continue (no text and no attachments) is invalid in single-assistant mode: the backend must return HTTP 400 before acquiring the Redis lease, and the frontend must keep the Send button disabled (via `_evalSendBtn()`) until text is typed or a file is attached. The gate is expressed via input-bar gate mode (`setGateMode`) — there is no separate gate panel widget. **Gate mode must survive page reload**: `chat_session_history.html` renders a `.chat-status-badge--gate[data-gate-context]` badge when `session.status == "awaiting_input"`; both the `DOMContentLoaded` bootstrap and the `htmx:afterSwap` handler scan for this badge and call `setGateMode()` to restore the Stop button and placeholder. **Send box always starts a new session when a restart panel is visible**: when `.chat-restart-panel` is present in the chat history, the `chatSendBtn` handler treats `activeSessionIdInput` as empty, forcing the create-session path — the user's text becomes the first message of a fresh run. The restart panel's own `data-session-id` attribute continues to own "Continue from last" / "Add context and continue" independently.
69. **`AgentMessageTermination` is mandatory for all team termination**: `MaxMessageTermination` from AutoGen must never be used in `agents/team_builder.py`. All termination must use `AgentMessageTermination` (defined in the same module), which counts only messages where `source != "user"` to avoid the off-by-one caused by the task message being included in AutoGen's built-in count. For gated runs, `AgentMessageTermination(n_agents) | ExternalTermination()` is required; the `ExternalTermination` instance must be stashed in `project["_runtime"]["external_termination"]` and registered via `runtime._EXTERNAL_TERMINATIONS[session_id]` so `cancel_team()` can call `.set()` for a graceful stop before the hard `CancellationToken.cancel()` fallback.
70. **`discussions[].content` must store raw user task only**: the `content` field on a human (`role=user`) discussion entry must be set to the raw user-typed text — never to `text_with_context` (the string that includes the full extracted PDF/DOCX/etc. attachment context block).
