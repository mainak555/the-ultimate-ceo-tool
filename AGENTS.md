## Key Rules

Canonical hierarchy for this repo:
- `docs/*.md` are the primary source of truth for architecture and behavior contracts.
- `.agents/skills/*/SKILL.md` are implementation checklists and enforcement guides.
- `README.md` is an onboarding summary and quick operator reference.

1. **Business logic lives in `server/services.py`** — views are thin controllers
2. **Validation lives in `server/schemas.py`** — returns clean dicts or raises `ValueError`
3. **MongoDB access lives in `server/db.py`** — singleton connection, no ORM
4. **Model catalog lives in `agent_models.json`** — model names are the UI key and are always displayed ascending
5. **AutoGen runtime code belongs in root `agents/`** — keep provider/client logic out of `server/`
6. **No Django ORM** — `DATABASES = {}`, sessions use signed cookies
7. **`APP_SECRET_KEY`** gates write access — HTMX requests send it as `X-App-Secret-Key`
8. **All env vars** read via `os.getenv()` with sensible defaults
9. **Templates** use HTMX partials pattern: full page loads `config.html`, subsequent interactions swap partials into `#main-content` or `#sidebar-list`
10. **SCSS** compiled by django-compressor + django-libsass
11. **No test suite yet** — planned for a future phase
12. **Project deletion safety**: never cascade delete chats when deleting a project. If any chat sessions exist for a project, deletion must be blocked with a clear error message.
13. **Common layer remains common**: global/shared modules (for example `server/static/server/js/app.js`) may contain only cross-feature utilities and hooks.
14. **Feature ownership is mandatory**: Home, Project Config, Trello, and Jira implementations must stay separated in HTMX templates, JS modules, views, and services. Avoid adding feature-specific logic to shared files.
15. **Provider registry is required for exports**: shared modules must use `server/static/server/js/provider_registry.js` (`window.ProviderRegistry`) instead of hardcoding provider names or provider-specific window globals.
16. **Reusable export modal pattern is mandatory**: all export providers (Trello, Jira, PDF, n8n, future) must use `window.ExportModalBase` (`server/static/server/js/export_modal_base.js`) as the shared modal shell. Providers implement adapters and must never build provider-local modal wrapper DOM. Canonical reference: `docs/UI.md` + `.agents/skills/export_popup_base/SKILL.md`.
17. **Visual consistency is mandatory across pages**: destructive controls (delete buttons/icons), color-token usage, spacing rhythm, and modal typography must match shared patterns defined in SCSS and docs; provider-specific theming is additive, not divergent.
18. **Extension skills are required for new providers**: follow `.agents/skills/export_popup_base/SKILL.md`, `.agents/skills/export_provider_adapter/SKILL.md`, `.agents/skills/ui_consistency_guardrails/SKILL.md`, `.agents/skills/scss_style_consistency/SKILL.md`, `.agents/skills/markdown_viewer_reuse/SKILL.md`, `.agents/skills/hierarchical_export_items/SKILL.md` (when items can nest), `.agents/skills/observability_logging/SKILL.md`, `.agents/skills/export_agents_sync/SKILL.md`, and `.agents/skills/shared_utility_reuse/SKILL.md` before implementing a new export provider.
19. **SCSS consistency is mandatory**: all styling changes must follow `docs/scss_style_guide.md`, use shared tokens/components, and must not introduce hardcoded color values when shared tokens exist.
20. **Markdown rendering must be reusable**: shared markdown rendering belongs in `server/static/server/js/markdown_viewer.js`; Home, Trello popup, Jira popup, and future providers must consume this common module instead of duplicating parsers.
21. **Jira export_agents are per-type**: for Jira, `export_agents` is scoped to each project type (`integrations.jira.software.export_agents`, etc.). There is no global `integrations.jira.export_agents` field. Validation, normalization, and template rendering must all read from the per-type config.
22. **Textarea fields in config forms must include a `<small class="form-hint">` below them** describing the field's purpose in plain language. The hint must be specific to the field's integration and type (e.g. "Prompt used by the extraction agent to parse the discussion into Jira Software issues."). Never leave a textarea without a hint.
23. **Nested fieldset indentation must be uniform across all nesting levels**: both L1 (`.form-group--nested`) and L2 (`.form-group--nested-l2`) use `margin-left: $space-md` and `padding-right: $space-sm`. Do not use `$space-lg` or any larger value for L2, and do not add `margin-top` to nested fieldsets — vertical rhythm is provided by the preceding element's `margin-bottom`. Missing `padding-right` causes textarea scrollbars and inputs to clip at the section edge. See `docs/scss_style_guide.md` §"Section Fieldsets (Config Form)" rules 4–6.
24. **`export_modal_base.js` is the only modal shell**: providers must call `window.ExportModalBase.open(ctx, adapter)` and implement the adapter interface in `.agents/skills/export_popup_base/SKILL.md`. No provider file may build an overlay DOM wrapper.
25. **Export modal context must include `projectId`**: the context object passed to `ProviderRegistry.openExportModal()` must always carry `{provider, sessionId, discussionId, secretKey, csrfToken, projectId}`. A missing `projectId` is a defect that must be fixed in `home.js`.
26. **Jira backend separation is mandatory**: `server/jira_service.py` is shared facade only; type-owned logic stays in `server/jira_software_service.py`, `server/jira_service_desk_service.py`, and `server/jira_business_service.py`.
27. **Jira frontend adapter separation is mandatory**: shared modal lifecycle logic stays in `server/static/server/js/jira_adapter_factory.js`; type-owned provider registration stays in `jira_software.js`, `jira_service_desk.js`, and `jira_business.js`.
28. **Jira layering skill is required for Jira refactors**: before Jira service/adapter refactors, follow `.agents/skills/jira_layer_separation/SKILL.md` and `docs/jira_integration.md`.
29. **Jira Software export modal metadata flow is mandatory**: project-specific dropdown options (issue type, priority, sprint, existing issue catalog) must come from `GET /jira/<session_id>/metadata/software/?project_key=<key>` and must be fetched via `jira_adapter_factory.js` after destination project selection.
30. **Jira Software global Sprint cascade is mandatory**: destination-level Sprint selector defines default values for all issue rows and must overwrite all rows whenever the global selector changes; row-level selectors remain editable for per-issue overrides after propagation.
31. **Jira Software fallback behavior is mandatory**: if project metadata cannot be loaded, the modal must remain usable by falling back to default issue type/priority options and minimal Sprint option (`Backlog`), with a non-blocking status message.
32. **Jira export label contract is mandatory**: `export_modal_base.js` header title uses adapter `label`, while export button text may use adapter `pushLabel` override. Jira Software currently uses header label `Jira` and push label `Jira Software`.
33. **Export popup visual contract is mandatory**: add-actions use `export-modal__context-add-btn`, editable cards use token-derived panel backgrounds, section headings use `export-modal__count-badge`, and provider connection states/messages remain consistent. Canonical visual contract: `docs/UI.md`, `docs/scss_style_guide.md`, `.agents/skills/export_popup_base/SKILL.md`, and `.agents/skills/ui_consistency_guardrails/SKILL.md`.
34. **Jira connection status message contract is mandatory**: connection rows must display `{Jira type label} Connected` on success and clear type-scoped error messages on failure.
35. **Jira metadata option deduplication is mandatory**: issue-type/priority options must be deduplicated by display label before rendering.
36. **Reserved numbering slot**: intentionally kept to preserve stable rule numbering for existing references.

38. **Project Config readonly markdown contract is mandatory**: readonly Objective, assistant system prompts, team selector system prompt, and integration extraction prompts (Trello and Jira types) must render via `window.MarkdownViewer.render()` using markdown target containers, not plain `<p>`/`<pre>` text blocks.
39. **Project Config readonly integrations parity is mandatory**: when integrations are enabled, readonly view must render Trello plus each enabled Jira type section (`software`, `service_desk`, `business`) with type-scoped readonly fields and extraction prompt visibility.
40. **Hierarchical export items use temp_id + parent_temp_id only**: any export provider whose items can nest (currently Jira Software, plus future providers) must persist parent linkage via `temp_id` + `parent_temp_id` and must NOT store `depth_level`. Depth is derived at render and push time. See `.agents/skills/hierarchical_export_items/SKILL.md` for the full data, render, and push contracts.
41. **Jira Software left-pane cascade is Project + Sprint only**: the Epic dropdown has been removed from the export modal. Parent linkage is expressed via the issue tree (`parent_temp_id`), not via a global Epic selector. The Sprint selector must always include a `Backlog` option whose value is the empty string.
42. **Jira Software push must be BFS with `temp_to_key` mapping**: `push_issues_software` walks roots first then breadth-first, populates `temp_to_key` as each issue is created, and resolves child `parent_temp_id` to a real Jira key before sending `fields.parent`. If an item carries `existing_issue_key`, push must update that existing Jira issue using the edited card fields, map `temp_to_key[temp_id] = existing_issue_key`, and continue BFS so descendants link correctly. A child whose parent failed to create must record a warning and be created as a root, never abort the batch. Result entries must echo `temp_id`.
43. **Jira Software sprint assignment via Agile API**: a non-empty `sprint` value triggers `POST /rest/agile/1.0/sprint/{id}/issue` AFTER the issue is created. An empty `sprint` value means Backlog and the Agile API call must be skipped entirely (the issue lands in backlog by default). Sprint failures are per-issue warnings, never batch failures. Issue types `Epic` and `Sub-task` are non-sprintable and must skip the Agile call with a warning.
44. **Observability contracts are mandatory for all I/O code**: modules performing HTTP, MongoDB, file, or LLM I/O must use module loggers (`logger = logging.getLogger(__name__)`), never use `print()`, never log secrets, and must propagate request IDs via `RequestIdMiddleware`. Full contract (event naming, redaction, payload handling, tracing decorators): `.agents/skills/observability_logging/SKILL.md` and `docs/observability.md`.
45. **OpenTelemetry is the project-wide tracing standard**: tracing uses the shared `core/tracing.py` contract with category toggles (`OTEL_INSTRUMENT_HTTP`, `OTEL_INSTRUMENT_MONGO`, `OTEL_INSTRUMENT_AGENTS`) and pluggable OTLP backend. Treat OTLP payloads as PII-bearing. Canonical architecture and env behavior: `docs/observability.md`.
46. **Model endpoint fallback contract is mandatory**: in `agents/factory.py`, endpoint resolution order is JSON `endpoint` first, then `{PROVIDER_UPPER}_API_URL` env fallback. API key resolution remains `{PROVIDER_UPPER}_API_KEY`. Azure providers (`azure_openai`, `azure_anthropic`) must still fail fast if endpoint is missing after fallback.
47. **Export schema documentation contract is mandatory**: every export popup provider must document its required extracted JSON schema in `README.md` (admin-facing) and in a provider-specific integration doc under `docs/` (developer-facing). Any schema or push-payload behavior change must update both docs in the same PR. Follow [docs/export_schema_contracts.md](docs/export_schema_contracts.md), [`.agents/skills/export_popup_base/SKILL.md`](.agents/skills/export_popup_base/SKILL.md), and [`.agents/skills/export_provider_adapter/SKILL.md`](.agents/skills/export_provider_adapter/SKILL.md).
48. **Shared HTTP span helper is mandatory for integrations**: outbound provider clients (Trello, Jira, and future integrations) must use `core/http_tracing.py` (`instrument_http_response`) for span enrichment and error status/details. Name response handlers meaningfully (for example `_handle_api_response`), never generic `_check`. Do not implement provider-local span plumbing helpers (`_current_span`, `_enrich_span`, `_mark_span_error`). `OTEL_HTTP_LOG_BODY` (default on) controls whether 2xx outbound calls capture `input.value`/`output.value`; non-2xx branches, including graceful fallback paths that continue execution, must still call `instrument_http_response(..., detail=...)` and always capture payloads/error details.
49. **MCP tool scope is per-agent + project-level shared**: each assistant agent stores `mcp_tools` ∈ {`none`, `shared`, `dedicated`}. `dedicated` requires non-empty per-agent `mcp_configuration`; `shared` requires non-empty project `shared_mcp_tools`. SSE transport is unsupported; only stdio and streamable HTTP are accepted. Canonical contract: `docs/mcp_integration.md` + `.agents/skills/mcp_tool_integration/SKILL.md`.
50. **MCP runtime wiring lives in `agents/mcp_tools.py`**: build/resolve workbenches in runtime layers only, attach with `reflect_on_tool_use=True`, and close via `runtime.evict_team()` -> `close_session_workbenches(session_id)`. Never construct MCP workbenches from `server/`.
51. **MCP secrets must never be logged or traced**: treat `args`, `env`, `headers`, and URL query params as secret-bearing. Logs/spans may expose only safe metadata (`scope`, `server_count`, `server_names`, `fingerprint`).
52. **MCP deployment topology is governed by `deployments/`**: `standalone/` bundles Node; `compose/` and `k8s/` use app + mcp-gateway sidecar over streamable HTTP.
53. **MCP secrets-by-reference is mandatory**: keep credentials in project `mcp_secrets` (UPPER_SNAKE keys), reference via `{KEY_NAME}` placeholders, and substitute at runtime only.
54. **MCP secrets follow the SECRET_MASK contract**: masked in form normalization, restored on save, and never rendered in readonly views.
55. **MCP fingerprint is computed over placeholders, not values**: keep fingerprints stable across secret rotations and credential-safe.
56. **Key/value form pattern is the only blessed UI for repeating KEY → value rows**: repeating config rows (MCP secrets, custom fields, future provider maps) must follow `.agents/skills/key_value_form_pattern/SKILL.md` (`1fr 2fr auto`, header add button, per-row delete, JS reindex, password + SECRET_MASK for sensitive values).
57. **Datetime storage standard is mandatory**: store Mongo datetimes as UTC BSON Date only (never preformatted strings), write via `_utc_now()`, coerce on read via `_coerce_dt_to_iso()`, and render via `<time class="local-time" data-utc="...">` + `window.renderLocalTimes()`. Canonical references: `docs/db_schema.md` and `.agents/skills/datetime_storage/SKILL.md`.
58. **Active run coordination is Redis-backed and fail-fast**: coordinate one run lease per `session_id` with heartbeat/cancel signaling; `POST /chat/sessions/<id>/run/` must fail fast when Redis is unavailable and must not enter `running`; MongoDB remains durable state for history/resume. Full lifecycle contract: `.agents/skills/active_session_coordination/SKILL.md` and `docs/agent_teams.md`.
59. **Run lifecycle refactors require the active-session skill**: follow `.agents/skills/active_session_coordination/SKILL.md` before changing run/start/stop/lease/cancel flow.
60. **Chat attachments are session-scoped by storage key**: all attachment blobs must use the flat key `sessions/{session_id}/attachments/{attachment_id}/{filename}` — no `message_id` segment, no project prefix. This key is assigned at upload time and is permanent (no rename at bind). All attachment lookup/bind/read operations must validate `session_id` equality before returning metadata or bytes.
61. **Session delete must clean attachment bytes, Redis cache, and metadata**: deleting a chat session must call `purge_session_attachment_cache(session_id)` first (purges Redis text-cache keys), then delete all blob objects under `sessions/{session_id}/`, then delete all `chat_attachments` metadata rows — in that order in the same operation.
62. **Attachment text extraction is lazy and Redis-backed**: `extracted_text` and `extraction_status` must never be stored in MongoDB. `server/attachment_service.py::build_attachment_context_block` checks Redis for cached text (`{REDIS_NAMESPACE}:attachment:{session_id}:{attachment_id}:text`) first; on miss it downloads from blob, extracts, writes to Redis with `REDIS_ATTACHMENT_TTL_SECONDS` TTL (default 24 h), and returns the full text. **Extraction failures (blob download error, pypdf error, etc.) are never written to Redis** — the next run retries extraction. Genuinely empty documents (e.g. scanned PDFs with no text layer) are cached normally to avoid repeated blob downloads. Supported extractable types: PDF (up to 50 pages), DOCX, PPTX (up to 50 slides), XLSX/XLS (all sheets, tab-separated rows), CSV (up to 200 rows), TXT, MD, JSON. Images are excluded from this path and loaded separately by `load_images_for_agents()`. Redis is provided by `agents.session_coordination.get_redis_client()` — `server/` code must never create a second Redis pool.
63. **Vision images are passed as `MultiModalMessage` to agents**: `server/views.py::_build_agent_task_for_run(task_text, session_id, attachment_ids)` downloads image bytes from blob via `load_images_for_agents()`, wraps each as `autogen_core.Image(PIL.Image.open(...))`, and returns `MultiModalMessage(content=[task_text, img1, img2, ...], source="user")` for `team.run_stream`. If PIL/autogen_core imports fail or all image downloads fail, it falls back to returning the plain task string — the agent run must never be blocked by image processing errors.
64. **Chat history attachment rendering contract is mandatory**: persisted chat messages must render attachment rows under message markdown content. Image attachments must render thumbnails; non-image attachments render filename links.
65. **HITL markdown + attachment parity is mandatory**: human gate notes must be markdown-rendered in live and persisted history, and the gate input surface must support the same attachment interactions as main chat input (attach button, drag/drop, paste).
66. **Chat attachment implementation skill is required**: before changing chat attachment upload/bind/render/delete behavior, follow [`.agents/skills/chat_attachment_workflow/SKILL.md`](.agents/skills/chat_attachment_workflow/SKILL.md).
67. **Azure attachment auth must use container SAS URL**: attachment storage auth must use `AZURE_STORAGE_CONTAINER_SAS_URL` (container-scoped SAS URL with query token), not account connection strings. SAS tokens should be least-privilege and short-lived where operationally possible.
68. **Single-assistant chat mode is mandatory**: when a project has exactly one assistant agent, the configuration must run in chat mode with Human Gate enabled and Team Setup hidden in the UI. Selector routing is invalid for one assistant. In this mode, runs pause after every assistant turn and continue until the human explicitly selects Stop (no max-iteration completion gate). Empty Continue (no text and no attachments) is invalid in single-assistant mode: the backend must return HTTP 400 before acquiring the Redis lease, and the frontend must keep the Send button disabled (via `_evalSendBtn()`) until text is typed or a file is attached. The gate is expressed via input-bar gate mode (`setGateMode`) — there is no separate gate panel widget. **Gate mode must survive page reload**: `chat_session_history.html` renders a `.chat-status-badge--gate[data-gate-context]` badge when `session.status == "awaiting_input"`; both the `DOMContentLoaded` bootstrap and the `htmx:afterSwap` handler scan for this badge and call `setGateMode()` to restore the Stop button and placeholder. **Send box always starts a new session when a restart panel is visible**: when `.chat-restart-panel` is present in the chat history, the `chatSendBtn` handler treats `activeSessionIdInput` as empty, forcing the create-session path — the user's text becomes the first message of a fresh run. The restart panel's own `data-session-id` attribute continues to own "Continue from last" / "Add context and continue" independently.
69. **`AgentMessageTermination` is mandatory for all team termination**: `MaxMessageTermination` from AutoGen must never be used in `agents/team_builder.py`. All termination must use `AgentMessageTermination` (defined in the same module), which counts only messages where `source != "user"` to avoid the off-by-one caused by the task message being included in AutoGen's built-in count. For gated runs, `AgentMessageTermination(n_agents) | ExternalTermination()` is required; the `ExternalTermination` instance must be stashed in `project["_runtime"]["external_termination"]` and registered via `runtime._EXTERNAL_TERMINATIONS[session_id]` so `cancel_team()` can call `.set()` for a graceful stop before the hard `CancellationToken.cancel()` fallback.
70. **`discussions[].content` must store raw user task only**: the `content` field on a human (`role=user`) discussion entry must be set to the raw user-typed text — never to `text_with_context` (the string that includes the full extracted PDF/DOCX/etc. attachment context block). Extracted attachment text is a runtime artefact assembled from Blob → Redis on each run; embedding it in MongoDB wastes document space, pollutes the export reference text consumed by Trello/Jira modals, and breaks the immutable source-of-truth contract for `discussions[].content`. In `server/views.py`, `text_with_context` is built solely for the agent task; `pending_messages[].content` must always be set to `task` (the raw input string).
71. **`MAX_AGENT_STATE_BYTES` is an env var and agent state persistence is non-fatal**: `config/settings.py` reads `MAX_AGENT_STATE_BYTES = int(os.getenv("MAX_AGENT_STATE_BYTES", "1000000"))` (default 1 MB). AutoGen serializes the full message history including base64 image bytes from `MultiModalMessage`, so image-heavy sessions can exceed the default — raise `MAX_AGENT_STATE_BYTES` in the environment for those deployments (MongoDB hard limit is 16 MB, shared with `discussions[]`). Agent state persistence is non-fatal: `checkpoint_state()` in `server/views.py` catches `ValueError` from `save_agent_state()` when the state exceeds the limit, logs `agents.session.state_too_large` at WARNING level, and lets the current run complete normally. Only session resume is unavailable when state cannot be persisted. `MAX_AGENT_STATE_BYTES` must appear in `config/settings.py` and in the README env var table.
72. **`export_agents` lists must reset to `[]` on any agent rename or removal**: when an assistant agent is renamed or deleted, every provider's `export_agents` allowlist (`integrations.trello.export_agents`, `integrations.jira.{software|service_desk|business}.export_agents`, and any future provider) must be silently reset to `[]` if it contains one or more names that no longer match the current agent roster — never raise a `ValueError` for this condition. An empty list means "show export on all agents", which is the safest fallback. This applies to all save paths including cloned projects. The frontend `syncExportAgentCheckboxes()` in `project_config.js` must sync every provider's checkbox-grid (Trello + all Jira types) whenever the agent list changes. Every provider's export_agents `checkbox-grid` div must carry a stable `id` so the JS can find and rebuild it. Follow [`.agents/skills/export_agents_sync/SKILL.md`](.agents/skills/export_agents_sync/SKILL.md) for the full backend validation pattern, frontend sync loop, template `id` requirements, and new-provider checklist.
73. **MCP OAuth 2.0 is mandatory for HTTP MCP servers requiring dynamic tokens**: `mcp_oauth_configs` is a project-level dict (`{server_name: {auth_url, token_url, client_id, client_secret, scopes?}}`), separate from `mcp_secrets`. `server_name` keys must exactly cross-validate against actual `mcpServers` keys (shared + dedicated) at save time — orphaned keys raise `ValueError`. `client_secret` uses the SECRET_MASK round-trip (stored masked, restored on save, never sent to browser). The flow uses Authorization Code + PKCE (S256). Callback URL `{BASE_URL}/mcp/oauth/callback/` must be registered with the OAuth provider. **Single generic start endpoint**: `GET /mcp/oauth/start/?flow=test|run&server_name=...&project_id=...&[session_id=...]&skey=...` is the only entry point — never add per-flow routes. Because popups (`window.open()`) cannot set custom headers, this endpoint accepts the admin secret as `?skey=` query param OR `X-App-Secret-Key` header (`_has_valid_oauth_secret()`); every other MCP endpoint stays header-only. Both success and error branches of `/mcp/oauth/callback/` MUST render the shared `server/templates/server/oauth_flow.html` template via `_render_outcome()` / `_render_error()` — never inline popup HTML in views. Auto-close timer on the error branch MUST be ≥ 30 s so users can read provider error messages. **Test Authorization** is config-form-only: opens with `?flow=test`, stores a 600 s status flag in Redis keyed by `project_id` (`{ns}:mcp_oauth:test:{project_id}:{server_name}:status`) — it does NOT write to the session token Redis key. **Run-time tokens** are session-scoped (`{ns}:mcp_oauth:run:{session_id}:{server_name}:token`); TTL is derived exclusively from the JWT `exp` claim: `TTL = exp − now(UTC)`. If the JWT cannot be decoded or `exp` is absent, falls back to a hardcoded 3 h default (`_MCP_OAUTH_DEFAULT_TTL`). No `expires_in` fallback. No env-var cap. The Redis key expires exactly when the Bearer token does, so cache hits during an active session are always valid. The pre-run gate is **server-driven**: `POST /chat/sessions/<id>/run/` calls `compute_pending_oauth_servers()` and returns **HTTP 409** `{status:"awaiting_oauth", servers:[...]}` when any server lacks a session-scoped Redis token. The frontend (`home.js::_doStartRun`) detects the 409 and calls `_showOAuthGatePanel()`, rendering an in-history `.chat-oauth-panel` card with per-server **Authorize** buttons — no client-side pre-check is performed. `window.McpOAuth` exposes only `openAuthPopup(serverName, sessionId, projectId, secretKey)` and `fetchStatus(sessionId, secretKey)` (3 s polling fallback); `checkAndAuthorize` does not exist. A **mid-run gate** also exists: if a token expires between turns the SSE stream emits an `awaiting_oauth` event, which triggers the same in-history panel; once all servers are re-authorized, `_onAllAuthorized()` replays the run automatically. No mid-session token refresh without user action (v1 limitation). **Observability**: every branch of start + callback emits a structured event (`agents.mcp.oauth_*`) and three nested OTel spans (`mcp.oauth.start`, `mcp.oauth.callback`, child `mcp.oauth.token_exchange`) carry `flow`, `server_name`, `http.status_code`, and (on non-2xx) a 500-char body snippet via `set_payload_attribute()`. Logs and span attributes MUST NEVER include `code`, `code_verifier`, `client_secret`, `access_token`, raw `?skey=` value, or full `Authorization` header. See [docs/mcp_integration.md](docs/mcp_integration.md) and [`.agents/skills/mcp_tool_integration/SKILL.md`](.agents/skills/mcp_tool_integration/SKILL.md) for the full OAuth section.
74. **Chat message copy button is always visible and uses raw markdown**: every chat bubble (user and agent) must carry a `.chat-bubble__copy-btn` in its `.chat-bubble__meta` row. The `data-raw-content` attribute on the bubble element holds the raw markdown from `discussions[].content` — never the rendered HTML. The click handler in `home.js` reads `data-raw-content`, appends a `**Attachments:**` markdown list for any `.chat-message-attachment__name` elements, and writes the combined text via `navigator.clipboard.writeText()` (with `execCommand("copy")` fallback). The copy button is **not** secret-gated — it must be visible and functional in both read and write sessions. The button transitions to a check-mark (`.chat-bubble__copy-btn--copied`, `$color-success`) for 2 s after a successful copy.
75. **Stop button must be disabled immediately on click**: the `chatStopBtn` click handler must set `chatStopBtn.disabled = true` before any async work (fetch or `sendRespond`) to prevent double-submission. A leading guard (`if (chatStopBtn.disabled) return;`) and synchronous validation checks (`sessionId`, `secretKey`) are allowed before the disable. In gate mode, if `sendRespond("stop")` rejects, `disabled` must be restored. `setRunningState()` must always reset `chatStopBtn.disabled = false` when hiding the button so the next run starts with a clean enabled state.
76. **Human Gate remote users config schema is mandatory**: `human_gate` contains `enabled`, `name`, `quorum` (`yes|first_win|team_config`, default `yes`), and `remote_users[]` (`{id,name,description}` with server-derived slug `id`). Names must be unique; per-user enable/disable is runtime-only. Single-assistant projects must reject non-empty `remote_users`. UI and validation contracts: `docs/human_gate_remote_users.md` + `.agents/skills/human_gate_remote_users/SKILL.md`.
77. **Human Gate remote-user readiness pre-run gate is mandatory**: when `remote_users` are configured, run-start must evaluate readiness before MCP OAuth and lease acquisition; unresolved users return HTTP 409 `awaiting_remote_users` and render the readiness panel flow. Keep token handling idempotent/TTL-bound and never log token strings or join URLs. Canonical endpoint/state contract: `docs/API.md` + `docs/human_gate_remote_users.md` + `.agents/skills/human_gate_remote_users/SKILL.md`.
78. **Remote-user export authorization uses delegated capability tokens**: remote-user pages must never receive `APP_SECRET_KEY`. Export modal calls from remote users are authorized by `X-Remote-Export-Capability` (session-scoped, short-lived Redis token bound to `{session_id, user_id}`). Session-scoped Trello/Jira endpoints must accept either `X-App-Secret-Key` (leader/admin path) or `X-Remote-Export-Capability` (remote path) and must validate that capability user belongs to configured `human_gate.remote_users` for the session's project. Logs and spans must include only safe attribution fields (`session_id`, `user_id`, provider/action) and must never include capability values.
79. **`run_stream()` must always receive a non-empty task**: in `server/views.py`, `effective_task` must never be falsy when passed to `team.run_stream()`. Anthropic Claude 3.7+ / Claude 4+ (and future strict providers) reject conversations whose last message is an `AssistantMessage` — a falsy task causes AutoGen to call `run_stream(task=None)`, producing an empty `UserMessage` list that triggers a 400 `invalid_request_error`. The fix is to unconditionally set `effective_task = "Continue."` whenever the resolved task is falsy, with **no `is_first_run` exception** (attachment text extraction can fail on the first run too). The synthetic `"Continue."` is not persisted to `discussions[]` and not shown in the SSE stream; it is only written into `agent_state` checkpoints. Follow [`.agents/skills/active_session_coordination/SKILL.md`](.agents/skills/active_session_coordination/SKILL.md) for the full guard contract and anti-patterns.
80. **Leader-aware quorum semantics are mandatory for Human Gate**: with `remote_users`, evaluate quorum with leader authority preserved: `yes` requires required remotes + leader continue, `first_win` accepts leader continue or first required remote response, and `team_config` requires targeted remote responders for that round. Only leader-controlled endpoints (`/respond/`, `/stop/`, `/run/`) may transition run state.
81. **MongoDB collection names must be constants-only**: runtime code must never hardcode collection-name literals (for example `"project_settings"`, `"chat_sessions"`, `"chat_attachments"`) in services, views, integrations, or runtime modules. Define collection constants in `server/db.py` and import those constants at every `get_collection(...)` call site. Before changing MongoDB access patterns, follow `.agents/skills/mongo_collection_constants/SKILL.md`.
82. **Utility/helper reuse is mandatory**: reusable pure helpers (time coercion, label normalization, confidence clamping, JSON serialization, auth guards) must live in shared utility modules and be imported by feature modules. Do not duplicate helper logic across views/services; extract shared helpers first, then call them from feature-owned code.
