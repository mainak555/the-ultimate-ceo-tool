## Key Rules

1. **Business logic lives in `server/services.py`** — views are thin controllers
2. **Validation lives in `server/schemas.py`** — returns clean dicts or raises `ValueError`
3. **MongoDB access lives in `server/db.py`** — singleton connection, no ORM
4. **Model catalog lives in `agent_models.json`** — model names are the UI key and are always displayed ascending
5. **AutoGen runtime code belongs in root `agents/`** — keep provider/client logic out of `server/`
6. **No Django ORM** — `DATABASES = {}`, sessions use signed cookies
7. **`APP_SECRET_KEY`** gates write access — HTMX requests send it as `X-App-Secret-Key`
8. **All env vars** read via `os.getenv()` with sensible defaults
9. **Templates** use HTMX partials pattern: full page loads `config.html`, subsequent interactions swap partials into `#main-content` or `#sidebar-list`
10. **SCSS** compiled by django-compressor + django-libsass
11. **No test suite yet** — planned for a future phase
12. **Project deletion safety**: never cascade delete chats when deleting a project. If any chat sessions exist for a project, deletion must be blocked with a clear error message.
12a. **Project version is server-managed**: the `version` float on every project document is set and conditionally incremented exclusively in `server/services.py` — never exposed as a form field and never accepted from user input. Rules: create → `1.0`; update → bumps `+0.1` **only** when (a) `team.type` changes OR (b) `human_gate.quorum` changes FROM `team_choice` to any other value — no other field changes trigger a bump, and multiple simultaneous conditions still produce exactly one `+0.1`; clone → `floor(source_version) + 1` (e.g. `1.x → 2.0`, `2.x → 3.0`). All update bump logic lives exclusively in `_compute_version_bump(existing, cleaned)` in `server/services.py` — extend this function to add new bump rules. `normalize_project()` defaults to `1.0` for legacy docs that lack the field. Display via `{{ project.version|floatformat:1 }}` in sidebar, config form header, and readonly view.
13. **Common layer remains common**: global/shared modules (for example `server/static/server/js/app.js`) may contain only cross-feature utilities and hooks.
14. **Feature ownership is mandatory**: Home, Project Config, Trello, and Jira implementations must stay separated in HTMX templates, JS modules, views, and services. Avoid adding feature-specific logic to shared files.
15. **Provider registry is required for exports**: shared modules must use `server/static/server/js/provider_registry.js` (`window.ProviderRegistry`) instead of hardcoding provider names or provider-specific window globals.
16. **Reusable export modal pattern is mandatory**: all export providers (Trello, Jira, PDF, n8n, future) must use `window.ExportModalBase` (`server/static/server/js/export_modal_base.js`) as the shared modal shell. The base owns the overlay, header ("Export to {label}"), 70/30 split layout, right-pane reference loading, and footer buttons. Providers implement an adapter object; never build modal DOM in provider files. For full adapter lifecycle, context contract, and base API usage, follow `.agents/skills/export_popup_base/SKILL.md` and `.agents/skills/export_provider_adapter/SKILL.md`.
17. **Visual consistency is mandatory across pages**: destructive controls (delete buttons/icons), color-token usage, spacing rhythm, and modal typography must match shared patterns defined in SCSS and docs; provider-specific theming is additive, not divergent.
18. **Extension skills are required for new providers**: follow `.agents/skills/export_popup_base/SKILL.md`, `.agents/skills/export_provider_adapter/SKILL.md`, `.agents/skills/ui_consistency_guardrails/SKILL.md`, `.agents/skills/scss_style_consistency/SKILL.md`, `.agents/skills/markdown_viewer_reuse/SKILL.md`, `.agents/skills/hierarchical_export_items/SKILL.md` (when items can nest), `.agents/skills/observability_logging/SKILL.md`, and `.agents/skills/export_agents_sync/SKILL.md` before implementing a new export provider.
19. **SCSS consistency is mandatory**: all styling changes must follow `docs/scss_style_guide.md` and must not introduce hardcoded color values when shared tokens exist.
20. **Markdown rendering must be reusable**: shared markdown rendering belongs in `server/static/server/js/markdown_viewer.js`; Home, Trello popup, Jira popup, and future providers must consume this common module instead of duplicating parsers.
21. **Jira export_agents are per-type**: for Jira, `export_agents` is scoped to each project type (`integrations.jira.software.export_agents`, etc.). There is no global `integrations.jira.export_agents` field. Validation, normalization, and template rendering must all read from the per-type config.
22. **Textarea fields in config forms must include a `<small class="form-hint">` below them** describing the field's purpose in plain language. The hint must be specific to the field's integration and type (e.g. "Prompt used by the extraction agent to parse the discussion into Jira Software issues."). Never leave a textarea without a hint.
23. **Nested fieldset indentation must be uniform across all nesting levels**: both L1 (`.form-group--nested`) and L2 (`.form-group--nested-l2`) use `margin-left: $space-md` and `padding-right: $space-sm`. Do not use `$space-lg` or any larger value for L2, and do not add `margin-top` to nested fieldsets — vertical rhythm is provided by the preceding element's `margin-bottom`. Missing `padding-right` causes textarea scrollbars and inputs to clip at the section edge. **`form-group--nested` is for fieldset sub-sections only — never for card-style rows** (e.g. MCP OAuth config rows, MCP Secrets rows): applying it to cards adds unwanted `margin-left` + `border-left` that visually indent the card away from the section title. Card rows must use only their dedicated BEM block class and must not carry `margin-left` at the row level. See `docs/scss_style_guide.md` §"Section Fieldsets (Config Form)" rules 4–7.
24. **`export_modal_base.js` is the only modal shell**: this is enforced by rule 16. Keep provider implementations adapter-only and use `.agents/skills/export_popup_base/SKILL.md` for the full contract.
25. **Export modal context must include `projectId`**: the context object passed to `ProviderRegistry.openExportModal()` must always carry `{provider, sessionId, discussionId, secretKey, csrfToken, projectId}`. A missing `projectId` is a defect that must be fixed in `home.js`.
26. **Jira backend and frontend separation is mandatory**: `server/jira_service.py` is the shared facade only; type-owned logic lives in `server/jira_software_service.py`, `server/jira_service_desk_service.py`, and `server/jira_business_service.py`. Frontend shared lifecycle logic belongs in `server/static/server/js/jira_adapter_factory.js` + `jira.js`; type-owned provider registration belongs only in `jira_software.js`, `jira_service_desk.js`, and `jira_business.js`.
27. **Jira layer ownership details live in skill/docs**: use `.agents/skills/jira_layer_separation/SKILL.md` for the implementation checklist and ownership boundaries.
28. **Jira refactors must use the layering skill**: before changing Jira services or export adapters, read `.agents/skills/jira_layer_separation/SKILL.md` and `docs/jira_integration.md`.
29. **Jira Software export modal metadata flow is mandatory**: project-specific dropdown options (issue type, priority, sprint, existing issue catalog) must come from `GET /jira/<session_id>/metadata/software/?project_key=<key>` and must be fetched via `jira_adapter_factory.js` after destination project selection.
30. **Jira Software global Sprint cascade is mandatory**: destination-level Sprint selector defines default values for all issue rows and must overwrite all rows whenever the global selector changes; row-level selectors remain editable for per-issue overrides after propagation.
31. **Jira Software fallback behavior is mandatory**: if project metadata cannot be loaded, the modal must remain usable by falling back to default issue type/priority options and minimal Sprint option (`Backlog`), with a non-blocking status message.
32. **Jira export label contract is mandatory**: `export_modal_base.js` header title uses adapter `label`, while export button text may use adapter `pushLabel` override. Jira Software currently uses header label `Jira` and push label `Jira Software`.
33. **Export popup add-action button style is mandatory**: follow `.agents/skills/ui_consistency_guardrails/SKILL.md` and `.agents/skills/scss_style_consistency/SKILL.md` for the exact class/token requirements.
34. **Export popup card background contract is mandatory**: follow `.agents/skills/ui_consistency_guardrails/SKILL.md` and `.agents/skills/scss_style_consistency/SKILL.md` for token-derived panel/background rules.
35. **Export popup item heading count badge is mandatory**: use the shared badge pattern from `.agents/skills/ui_consistency_guardrails/SKILL.md` so counts remain consistent across providers.
36. **Jira connection status message contract is mandatory**: the Jira export modal connection row must display `{Jira type label} Connected` on success (for example `Jira Software Connected`) and a clear type-scoped error message on failure.
37. **Jira metadata option deduplication is mandatory**: issue-type/priority dropdown options must be deduplicated by display label before rendering so duplicate labels (for example `Epic`) are never shown twice in issue cards.
38. **Project Config readonly markdown contract is mandatory**: readonly Objective, assistant system prompts, team selector system prompt, and integration extraction prompts (Trello and Jira types) must render via `window.MarkdownViewer.render()` using markdown target containers, not plain `<p>`/`<pre>` text blocks.
39. **Project Config readonly integrations parity is mandatory**: when integrations are enabled, readonly view must render Trello plus each enabled Jira type section (`software`, `service_desk`, `business`) with type-scoped readonly fields and extraction prompt visibility.
40. **Hierarchical export items use temp_id + parent_temp_id only**: any export provider whose items can nest (currently Jira Software, plus future providers) must persist parent linkage via `temp_id` + `parent_temp_id` and must NOT store `depth_level`. Depth is derived at render and push time. See `.agents/skills/hierarchical_export_items/SKILL.md` for the full data, render, and push contracts.
41. **Jira Software left-pane cascade is Project + Sprint only**: the Epic dropdown has been removed from the export modal. Parent linkage is expressed via the issue tree (`parent_temp_id`), not via a global Epic selector. The Sprint selector must always include a `Backlog` option whose value is the empty string.
42. **Jira Software push must be BFS with `temp_to_key` mapping**: `push_issues_software` walks roots first then breadth-first, populates `temp_to_key` as each issue is created, and resolves child `parent_temp_id` to a real Jira key before sending `fields.parent`. If an item carries `existing_issue_key`, push must update that existing Jira issue using the edited card fields, map `temp_to_key[temp_id] = existing_issue_key`, and continue BFS so descendants link correctly. A child whose parent failed to create must record a warning and be created as a root, never abort the batch. Result entries must echo `temp_id`.
43. **Jira Software sprint assignment via Agile API**: a non-empty `sprint` value triggers `POST /rest/agile/1.0/sprint/{id}/issue` AFTER the issue is created. An empty `sprint` value means Backlog and the Agile API call must be skipped entirely (the issue lands in backlog by default). Sprint failures are per-issue warnings, never batch failures. Issue types `Epic` and `Sub-task` are non-sprintable and must skip the Agile call with a warning.
44. **Observability contracts are mandatory for all I/O code**: every Python module that performs HTTP, MongoDB, file, or LLM I/O must follow `.agents/skills/observability_logging/SKILL.md` for logger, event naming, redaction, payload, and tracing decorator contracts.
45. **OpenTelemetry is the project-wide tracing standard**: all I/O tracing uses `@traced_function` / `traced_block` from `core/tracing.py`, with category-gated instrumentation toggles and pluggable OTLP export. Use `docs/observability.md` for env-var defaults, instrumentation matrix, and rollout/validation guidance.
46. **Model endpoint fallback contract is mandatory**: in `agents/factory.py`, endpoint resolution order is JSON `endpoint` first, then `{PROVIDER_UPPER}_API_URL` env fallback. API key resolution remains `{PROVIDER_UPPER}_API_KEY`. Azure providers (`azure_openai`, `azure_anthropic`) must still fail fast if endpoint is missing after fallback.
47. **Export schema documentation contract is mandatory**: every export popup provider must document its required extracted JSON schema in `README.md` (admin-facing) and in a provider-specific integration doc under `docs/` (developer-facing). Any schema or push-payload behavior change must update both docs in the same PR. Follow [docs/export_schema_contracts.md](docs/export_schema_contracts.md), [`.agents/skills/export_popup_base/SKILL.md`](.agents/skills/export_popup_base/SKILL.md), and [`.agents/skills/export_provider_adapter/SKILL.md`](.agents/skills/export_provider_adapter/SKILL.md). The modal context passed to `ProviderRegistry.openExportModal()` must include `projectId` per rule 25.
48. **Shared HTTP span helper is mandatory for integrations**: outbound provider clients (Trello, Jira, and future integrations) must use `core/http_tracing.py` (`instrument_http_response`) for span enrichment and error status/details. Name response handlers meaningfully (for example `_handle_api_response`), never generic `_check`. Do not implement provider-local span plumbing helpers (`_current_span`, `_enrich_span`, `_mark_span_error`). `OTEL_HTTP_LOG_BODY` (default on) controls whether 2xx outbound calls capture `input.value`/`output.value`; non-2xx branches, including graceful fallback paths that continue execution, must still call `instrument_http_response(..., detail=...)` and always capture payloads/error details.
49. **MCP tool scope is per-agent + project-level shared**: each assistant agent stores `mcp_tools` in {`none`, `shared`, `dedicated`}; `dedicated` requires non-empty agent `mcp_configuration`, and `shared` requires non-empty project `shared_mcp_tools`. SSE transport is not supported; use only stdio or streamable HTTP. Full schema and validation rules live in `.agents/skills/mcp_tool_integration/SKILL.md` and `docs/mcp_integration.md`.
50. **MCP runtime wiring lives in `agents/mcp_tools.py`**: build workbenches via `resolve_mcp_servers_for_agent()` and `build_mcp_workbenches()`, pass to `AssistantAgent(..., reflect_on_tool_use=True)`, and clean up in `agents/runtime.py::evict_team()` via `close_session_workbenches(session_id)`. See the MCP skill for full lifecycle and thread-safety constraints.
51. **MCP secrets must never be logged or traced**: treat MCP server `args`, `env`, `headers`, and URL query strings as credentials. Secret redaction and safe span attribute contracts are defined in `.agents/skills/mcp_tool_integration/SKILL.md` and `docs/observability.md`.
52. **MCP deployment topology is governed by `deployments/`**: standalone bundles Node in app image; compose and k8s use a Python app plus Node mcp-gateway sidecar over streamable HTTP. See `docs/mcp_integration.md` for deployment contracts.
53. **MCP secrets-by-reference is mandatory for all credential material**: never embed raw secrets in `mcp_configuration` or `shared_mcp_tools`; store under project `mcp_secrets` and reference via `{KEY_NAME}` placeholders. Follow `.agents/skills/mcp_tool_integration/SKILL.md` for key format validation, placeholder resolution, and save-path behavior.
54. **MCP secrets follow the SECRET_MASK contract**: form re-renders must mask stored values and restore only when the user keeps them unchanged. Readonly views must not expose secret values. See `.agents/skills/mcp_tool_integration/SKILL.md` for exact masking/restore behavior.
55. **MCP secret fingerprint is computed over placeholders, not values**: keep fingerprints stable across secret rotation and never include credential material in logs/spans. See `.agents/skills/mcp_tool_integration/SKILL.md` for the exact fingerprint contract.
56. **Key/value form pattern is the only blessed UI for repeating KEY → value rows**: MCP Secrets, Trello Custom Fields, and any future provider list-of-pairs control (Jira custom fields, n8n inputs, header maps) must follow [`.agents/skills/key_value_form_pattern/SKILL.md`](.agents/skills/key_value_form_pattern/SKILL.md) — `1fr 2fr auto` grid, `+ Add` button in the subsection header, `chat-session-item__delete` × button per row, JS reindex on remove, and (for sensitive values) `<input type="password">` with the SECRET_MASK round-trip.
57. **Datetime storage standard is mandatory**: store MongoDB datetimes as BSON Date values (never ISO strings or bare `HH:MM`), use `util.utc_now()` for writes, coerce reads via `_coerce_dt_to_iso()`, and render through `<time class="local-time" data-utc="ISO">` + `window.renderLocalTimes()`. See `.agents/skills/datetime_storage/SKILL.md` for the full field checklist and implementation patterns.
58. **Active run coordination is Redis-backed and fail-fast**: chat run ownership must be coordinated by `agents/session_coordination.py` using one lease per `session_id` plus heartbeat and cross-instance cancel signaling. `POST /chat/sessions/<id>/run/` must fail fast when Redis is unavailable and must not enter `running`. MongoDB remains the durable source for `chat_sessions.agent_state` and discussion history used for resume.
59. **Run lifecycle refactors require the active-session skill**: before changing chat run/start/stop/lease/cancel behavior, follow [`.agents/skills/active_session_coordination/SKILL.md`](.agents/skills/active_session_coordination/SKILL.md).
60. **Remote user quorum changes require the remote-user-quorum skill**: before adding a new quorum mode, changing `chat_session_respond` or `remote_user_respond` quorum logic, modifying quorum event fanout, or changing gate Redis helpers, follow [`.agents/skills/remote_user_quorum/SKILL.md`](.agents/skills/remote_user_quorum/SKILL.md). `remote_users` and `quorum` are always read from the live project — never stored in `chat_sessions`.
60a. **Chat surface contracts are shared across Home/HITL, Remote, and Guest**: shared chat classes and bubble DOM structure must remain aligned across all chat surfaces; Remote and Guest public pages must keep a unified header contract (left title, right role badge). Send + attachment interactions must remain consistent between Home/HITL and Remote, while Guest remains readonly. Before modifying these surfaces, follow [`.agents/skills/chat_surface_shared/SKILL.md`](.agents/skills/chat_surface_shared/SKILL.md) and [`.agents/skills/chat_compose_attachment_contract/SKILL.md`](.agents/skills/chat_compose_attachment_contract/SKILL.md).
60b. **`QUORUM_OPTIONS` has a single source of truth in `server/util.py`**: option values and labels are defined once there; imported by `views.py` (config form context + `window._quorumOptions` JSON), and by `remote_user_views.py` (`VALID_QUORUM_VALUES`). Never hardcode quorum option arrays in templates or JS. **`team_choice` must never appear as a selectable option** in the live participants readiness dropdown — it is only shown (disabled) when the project is already configured with that value. The server-rendered `.chat-remote-panel` must carry `data-quorum="{{ session_quorum|default:'na' }}"` so the dropdown renders immediately on page load and session switch without waiting for the WebSocket `state` message. Resolution order for `session_quorum`: Redis quorum key → project config quorum → `"na"`.
60c. **Host session websocket continuity is mandatory**: `home.js` must keep `/ws/session/<session_id>/` connected for active sessions, including newly created sessions. `first_win` auto-resume depends on receiving `quorum_committed`, and remote-user chat bubbles depend on `message` events from this channel.
60d. **Remote disconnect deferred-readiness latch is mandatory**: when a configured remote participant disconnects during `session.status == "running"`, the current run must continue uninterrupted, but the next `POST /chat/sessions/<id>/run/` must be blocked behind the existing `.chat-remote-panel` readiness gate unless required participants are satisfied (online or ignored). This latch applies only to configured remote users, never guest watchers, and must not trigger for users already marked `ignored`. Latch TTL uses `REDIS_REMOTE_USER_TOKEN_TTL_SECONDS`; clear it when readiness is satisfied or on session-key purge.
60e. **Home project context participants + quorum chip contract is mandatory**: in `server/templates/server/partials/chat_session_history.html` (project context state), the Participants section must render assistant agents, the human gate owner (when enabled), and all configured `human_gate.remote_users`. The team metadata row must render as separate chips: base chip `team.type + max_iterations`, plus a distinct quorum chip shown only when `remote_users` is non-empty. The quorum chip text must resolve from `server/util.py::QUORUM_OPTIONS` labels (for example via template filter), never hardcoded in templates/JS.
61. **Chat attachments are session-scoped by storage key**: all attachment blobs must use the flat key `sessions/{session_id}/attachments/{attachment_id}/{filename}` — no `message_id` segment, no project prefix. This key is assigned at upload time and is permanent (no rename at bind). All attachment lookup/bind/read operations must validate `session_id` equality before returning metadata or bytes.
62. **Session delete must clean attachment bytes, Redis cache, and metadata**: deleting a chat session must call `purge_session_attachment_cache(session_id)` first (purges Redis text-cache keys), then delete all blob objects under `sessions/{session_id}/`, then delete all `chat_attachments` metadata rows — in that order in the same operation.
63. **Attachment text extraction is lazy and Redis-backed**: `extracted_text` and `extraction_status` must never be stored in MongoDB. `server/attachment_service.py::build_attachment_context_block` checks Redis for cached text (`{REDIS_NAMESPACE}:attachment:{session_id}:{attachment_id}:text`) first; on miss it downloads from blob, extracts, writes to Redis with `REDIS_ATTACHMENT_TTL_SECONDS` TTL (default 24 h), and returns the full text. **Extraction failures (blob download error, pypdf error, etc.) are never written to Redis** — the next run retries extraction. Genuinely empty documents (e.g. scanned PDFs with no text layer) are cached normally to avoid repeated blob downloads. Supported extractable types: PDF (up to 50 pages), DOCX, PPTX (up to 50 slides), XLSX/XLS (all sheets, tab-separated rows), CSV (up to 200 rows), TXT, MD, JSON. Images are excluded from this path and loaded separately by `load_images_for_agents()`. Redis is provided by `agents.session_coordination.get_redis_client()` — `server/` code must never create a second Redis pool.
64. **Vision images are passed as `MultiModalMessage` to agents**: `server/views.py::_build_agent_task_for_run(task_text, session_id, attachment_ids)` downloads image bytes from blob via `load_images_for_agents()`, wraps each as `autogen_core.Image(PIL.Image.open(...))`, and returns `MultiModalMessage(content=[task_text, img1, img2, ...], source="user")` for `team.run_stream`. If PIL/autogen_core imports fail or all image downloads fail, it falls back to returning the plain task string — the agent run must never be blocked by image processing errors.
65. **Chat history attachment rendering contract is mandatory**: persisted chat messages must render attachment rows under message markdown content. Image attachments must render thumbnails; non-image attachments render filename links.
66. **HITL markdown + attachment parity is mandatory**: human gate notes must be markdown-rendered in live and persisted history, and the gate input surface must support the same attachment interactions as main chat input (attach button, drag/drop, paste).

69. **Single-assistant chat mode is mandatory**: when a project has exactly one assistant agent **and no remote users are configured**, the configuration must run in chat mode with Human Gate enabled and Team Setup hidden in the UI. Selector routing is invalid for one assistant. In this mode, runs pause after every assistant turn and continue until the human explicitly selects Stop (no max-iteration completion gate). Empty Continue (no text and no attachments) is invalid in single-assistant mode: the backend must return HTTP 400 before acquiring the Redis lease, and the frontend must keep the Send button disabled (via `_evalSendBtn()`) until text is typed or a file is attached. **Exception — single-assistant with remote users**: when exactly one assistant is configured but at least one remote user is present, the project exits pure single-assistant chat mode: Team Setup becomes visible and its values are honored, `max_iterations` governs run completion, and empty Continue is allowed (same rules as multi-assistant). Human Gate is still forced on. Selector team type remains invalid regardless of remote users (selector requires ≥ 2 agents). The gate is expressed via input-bar gate mode (`setGateMode`) — there is no separate gate panel widget. **Gate mode must survive page reload**: `chat_session_history.html` renders a `.chat-status-badge--gate[data-gate-context]` badge when `session.status == "awaiting_input"`; both the `DOMContentLoaded` bootstrap and the `htmx:afterSwap` handler scan for this badge and call `setGateMode()` to restore the Stop button and placeholder. **Send box always starts a new session when a restart panel is visible**: when `.chat-restart-panel` is present in the chat history, the `chatSendBtn` handler treats `activeSessionIdInput` as empty, forcing the create-session path — the user's text becomes the first message of a fresh run. The restart panel's own `data-session-id` attribute continues to own "Continue from last" / "Add context and continue" independently.
70. **`AgentMessageTermination` is mandatory for all team termination**: `MaxMessageTermination` from AutoGen must never be used in `agents/team_builder.py`. For gated runs, `AgentMessageTermination(n_agents) | ExternalTermination()` is required; the `ExternalTermination` instance must be stashed in `project["_runtime"]["external_termination"]` and registered via `runtime._EXTERNAL_TERMINATIONS[session_id]` so `cancel_team()` can call `.set()` for a graceful stop before the hard `CancellationToken.cancel()` fallback.
71. **`discussions[].content` must store raw user task only**: the `content` field on a human (`role=user`) discussion entry must be set to the raw user-typed text — never to `text_with_context` (the string that includes the full extracted PDF/DOCX/etc. attachment context block). Extracted attachment text is a runtime artefact assembled from Blob → Redis on each run; embedding it in MongoDB wastes document space, pollutes the export reference text consumed by Trello/Jira modals, and breaks the immutable source-of-truth contract for `discussions[].content`. In `server/views.py`, `text_with_context` is built solely for the agent task; `pending_messages[].content` must always be set to `task` (the raw input string).
72. **`MAX_AGENT_STATE_BYTES` is an env var and agent state persistence is non-fatal**: `config/settings.py` reads `MAX_AGENT_STATE_BYTES = int(os.getenv("MAX_AGENT_STATE_BYTES", "1000000"))` (default 1 MB). AutoGen serializes the full message history including base64 image bytes from `MultiModalMessage`, so image-heavy sessions can exceed the default — raise `MAX_AGENT_STATE_BYTES` in the environment for those deployments (MongoDB hard limit is 16 MB, shared with `discussions[]`). Agent state persistence is non-fatal: `checkpoint_state()` in `server/views.py` catches `ValueError` from `save_agent_state()` when the state exceeds the limit, logs `agents.session.state_too_large` at WARNING level, and lets the current run complete normally. Only session resume is unavailable when state cannot be persisted. `MAX_AGENT_STATE_BYTES` must appear in `config/settings.py` and in the README env var table.

86. **Anthropic error handling is owned exclusively by `_RetryAnthropicClient`**: the proxy in `agents/factory.py` provides two guards applied before every `create()` / `create_stream()` call — never add either inline in `team_builder.py`, `runtime.py`, or agent code. (a) **529 retry**: transient `OverloadedError` (HTTP 529) is retried via exponential backoff with jitter (`ANTHROPIC_MAX_RETRIES` attempts, `ANTHROPIC_RETRY_BASE_DELAY` base seconds, defaults `2` and `5.0`). All other errors propagate immediately. (b) **Prefill guard** (`_ensure_user_message_last`): Claude 4+ (and later Anthropic models) reject API calls whose message list ends with an `AssistantMessage` ("does not support assistant message prefill"). This can happen in group-chat rounds where an agent's AutoGen message buffer is empty when its turn fires, leaving the model context ending with its own prior response. The guard appends a minimal synthetic `UserMessage("Please continue.", source="user")` to the messages list before forwarding to the inner client when the last message is an `AssistantMessage`. The synthetic message is invisible to the rest of the application (not persisted to `discussions[]`, not shown in the chat stream). Both `anthropic` and `azure_anthropic` providers are wrapped automatically by their respective builder functions.
